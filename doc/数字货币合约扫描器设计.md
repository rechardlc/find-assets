# 数字货币合约扫描器设计

## 1. 背景

当前项目已经实现 A 股扫描器，核心能力包括：

- A 股清单拉取与本地日缓存
- 日线 / 周线周期抽象
- `pierce` / `reversal` 公共策略
- CLI 与 HTTP API 扫描入口
- JSON / Markdown / Console 导出

新增数字货币合约扫描器时，应保持 A 股入口和行为不变。数字货币市场独立运行，新增专用命令入口，复用已有指标与策略能力。

## 2. 目标

新增一个独立的 USDT 永续合约扫描器：

- 新增 `cmd/crypto-scanner`，不改动现有 `cmd/scanner` 的默认 A 股行为
- 优先使用 Binance USDT 永续合约数据，失败后回退 OKX USDT 永续合约数据
- 每天第一次运行时生成热门山寨合约池缓存，同日后续运行复用缓存
- 合约池默认使用 `hot_alt` 综合热度，不单纯按交易量排序
- 每 15 分钟对齐 K 线收盘后延迟扫描，默认延迟 20 秒
- 扫描策略复用现有 `reversal` 超跌拐点逻辑
- 输出沿用现有 exporter 体系
- 命中结果时默认通过 QQ 邮箱 SMTP 推送到 `richard_0525@foxmail.com`

## 3. 非目标

第一版不做以下事情：

- 不接交易账户，不下单
- 不做浏览器或 App 热搜抓取
- 不做 Web UI
- 不改造现有 A 股 HTTP API
- 不做盘中自动刷新合约池
- 不引入数据库

## 4. 运行方式

示例：

```powershell
crypto-scanner.exe
```

不带参数时等价于：

```powershell
crypto-scanner.exe -source binance,okx -pool hot_alt -top 20 -interval 15m -pattern reversal -schedule -delay 20s
```

参数：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `source` | `binance,okx` | 数字货币数据源回退顺序 |
| `pool` | `hot_alt` | 合约池规则 |
| `top` | `20` | 每日缓存的候选合约数量 |
| `interval` | `15m` | K 线周期 |
| `pattern` | `reversal` | 策略形态 |
| `bars` | `300` | 每个合约拉取的 K 线数量 |
| `schedule` | `true` | 是否按周期持续扫描；单次扫描可传 `-schedule=false` |
| `delay` | `20s` | K 线收盘后的延迟执行时间 |
| `export` | `console` | 导出格式，复用 `console,json,md` |
| `out` | `./output` | 文件导出目录 |
| `mail` | `true` | 命中时发送邮件通知 |
| `mail-to` | `richard_0525@foxmail.com` | 邮件收件人 |
| `mail-from` | `richard_0525@foxmail.com` | 邮件发件人 |
| `smtp-host` | `smtp.qq.com` | SMTP 服务器 |
| `smtp-port` | `465` | SMTP 端口 |
| `smtp-user` | `richard_0525@foxmail.com` | SMTP 用户名 |
| `smtp-pass` | `FIND_ASSETS_SMTP_PASS` | SMTP 授权码，默认从 `.env` 或系统环境变量读取 |
| `env` | `.env` | 环境变量文件路径 |

## 5. 合约池规则

第一版实现 `hot_alt` 池，用公开接口可获取的字段计算综合热度。

过滤条件：

- 只保留 USDT 永续合约
- 只保留交易状态正常的合约
- 排除 BTC、ETH
- 排除稳定币或明显非山寨标的

综合分：

```text
hot_score =
  amplitude_score
+ abs_change_score
+ quote_volume_score
+ funding_abs_score
```

字段来源：

| 指标 | Binance | OKX |
|------|---------|-----|
| 24h 高低价 / 涨跌幅 / 成交额 | `/fapi/v1/ticker/24hr` | `/api/v5/market/tickers?instType=SWAP` |
| 当前资金费率 | `/fapi/v1/premiumIndex` | 第一版暂未逐合约接入，后续补 `/api/v5/public/funding-rate` |
| 15m K 线 | `/fapi/v1/klines` | `/api/v5/market/candles` |

振幅计算：

```text
amplitude = (high24h - low24h) / open24h * 100
```

资金费率异常：

```text
funding_abs = abs(current_funding_rate)
```

超跌策略可重点关注负资金费率，表示空头拥挤，但第一版综合热度先使用绝对异常值。

## 6. 缓存策略

每天第一次检索合约池时写入本地缓存：

```text
crypto/
  pools/
    hot_alt_20260617_binance.json
```

缓存内容：

```json
{
  "date": "2026-06-17",
  "exchange": "binance",
  "pool": "hot_alt",
  "contract": "usdt_perp",
  "top": 20,
  "generated_at": "2026-06-17T10:15:20+08:00",
  "assets": [
    {
      "symbol": "1000PEPEUSDT",
      "name": "1000PEPE USDT Perpetual",
      "exchange": "binance",
      "score": 86.3,
      "price_change_percent": -8.2,
      "amplitude": 18.5,
      "quote_volume": 123456789,
      "funding_rate": -0.0003
    }
  ]
}
```

同日有缓存时直接读取缓存，不再重新扫描全市场热门池。每 15 分钟扫描只拉缓存合约的 15m K 线。

## 7. 数据源回退

数据源顺序默认 `binance,okx`：

1. 优先从 Binance 拉取热门池和 K 线
2. Binance 热门池失败时，整体回退 OKX
3. Binance 单个 K 线失败时，可对该合约尝试 OKX 符号映射后回退
4. 两个数据源均失败时，本轮扫描失败并输出错误

内部统一合约标识：

| 内部字段 | Binance | OKX |
|----------|---------|-----|
| `Symbol` | `BTCUSDT` | 由 `BTC-USDT-SWAP` 归一化为 `BTCUSDT` |
| `ExchangeSymbol` | `BTCUSDT` | `BTC-USDT-SWAP` |
| `Exchange` | `binance` | `okx` |

## 8. 定时扫描

默认启用 `-schedule` 持续运行：

```text
扫描时间 = 下一个 15 分钟边界 + delay
```

示例：

```text
09:00:20
09:15:20
09:30:20
09:45:20
```

如果上一轮扫描尚未结束，则跳过当前轮，避免并发请求和重复输出。

## 10. 模块划分

新增模块：

```text
cmd/crypto-scanner/
internal/crypto/
```

建议文件：

| 文件 | 职责 |
|------|------|
| `cmd/crypto-scanner/main.go` | CLI 入口、参数解析、调度 |
| `internal/crypto/model.go` | 合约资产、行情指标、缓存结构 |

| `internal/crypto/source.go` | 数字货币数据源接口与回退组合 |
| `internal/crypto/binance.go` | Binance USDT 永续公开接口 |
| `internal/crypto/okx.go` | OKX USDT 永续公开接口 |
| `internal/crypto/service.go` | 扫描编排，复用 strategy 和 exporter |

## 11. 测试策略

优先覆盖纯逻辑：

- `hot_alt` 评分排序和主流币过滤
- Binance / OKX 符号归一化
- 每日缓存命名、读取、过期处理
- 15m 边界延迟时间计算
- 数字货币扫描服务能复用 `strategy.Get("15m", "reversal", ...)`

网络接口通过 `httptest.Server` 或注入 HTTP client 测试响应解析，不在单元测试中访问真实交易所。

## 12. 风险

| 风险 | 应对 |
|------|------|
| Binance 网络不可用 | 默认回退 OKX |
| 交易所字段变化 | 接口解析失败时返回明确错误 |
| 热门池一天内过时 | 第一版接受，后续可加 `refresh-pool` |
| 当前 15m K 线未收盘 | 对齐边界后延迟执行 |
| 山寨币波动过大 | 输出信号，不做自动交易 |
| SMTP 授权码泄露 | 不写入代码或文档，优先使用环境变量 |
